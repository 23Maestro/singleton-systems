## Recruiting Funnel Boilerplate

Use recruiting as the reference shape because it exposes every paid-growth problem clearly:

```text
1. Audience has a high-emotion goal.
2. Buyer is not always the same person as the visible user.
3. Bad leads are cheaper than good leads.
4. The funnel must teach the ad platform which lead is valuable.
5. Trust must be built before asking for personal data.
6. Sales follow-up must be fast, specific, and tied to the original promise.
7. Revenue tracking must separate new acquisition from recurring revenue.
```

Generic version for any client:

```text
Visible user != economic buyer
Cheap engagement != qualified demand
Lead volume != pipeline value
Ad clicks != workflow readiness
Portfolio assets != portfolio system
```

## Core Logic

### 1. Grade leads before optimizing

Do not optimize paid traffic for every form submit.

Use a simple event/value split:

```text
High-value buyer signal -> positive value
Low-intent or wrong-fit signal -> zero or low value
Purchase signal -> actual transaction value
```

This prevents the platform from finding the cheapest people to convert when those people do not buy.

### 2. Start with faster signal, then move to money signal

Phase 1:

```text
Goal: train the platform on qualified leads.
Target: enough qualified lead events to stabilize delivery.
Metric: cost per qualified lead.
```

Phase 2:

```text
Goal: optimize toward revenue.
Trigger: enough purchase events to trust value optimization.
Metric: CAC, ROAS, payback.
```

Do not force purchase optimization before the account has enough purchase data. It can starve delivery or teach from thin signal.

### 3. Creative is targeting

Creative does not just "sell." It filters.

Reusable filters:

```text
Direct call-out: name the buyer, role, pain, or situation.
Contextual signal: use words only the right buyer recognizes.
Portfolio cue: show credible people, assets, outcomes, or process.
Workflow demo: show how the thing works, not just the promise.
Wrong-fit repellant: make bad-fit users self-select out.
```

### 4. Portfolio must match the offer

Portfolio should answer:

```text
Why should I trust you?
What changes if I act?
What happens after I submit?
Who is behind this?
What makes this different from generic alternatives?
```

Strong portfolio types:

```text
short expert/founder video
specific process walkthrough
before/after workflow map
testimonial with visible context
logos or recognizable trust markers
screenshots of the actual operating system
data dashboard screenshot with sensitive values removed
```

Priority portfolio rule:

```text
Show the work moving through the system before asking people to believe the system is valuable.
```

### 5. Bridge before ask

If the page asks for personal data too early, paid traffic leaks.

Better sequence:

```text
Ad promise
Bridge page or value section
Trust/portfolio
Specific mechanism
Lead capture
Next-step expectation
```

The bridge page does not need to be complex. It needs to increase trust before the form.

### 6. Keep the refresh loop simple

Minimum creative loop:

```text
Produce 25-35 short vertical pieces.
Post 3-5 per day for 7-10 days.
Track hook, hold, comments, and audience match.
Promote only the top 3-4 into paid.
Keep winners.
Replace losers.
Let ads run at least 72 hours before judging.
Prefer 7 days when spend allows.
```

Do not over-read one day of performance. Early ad data is noisy.

## Useful Benchmarks

Use these as directional checks, not universal guarantees:

```text
Hook rate: strong if above 25%; weak if below 20%.
Hold rate: strong if above 10-15%; weak if people drop immediately.
Paid CTR: workable if above 1%; weak if below 0.5%.
CPC: investigate if traffic costs are far above normal for the market.
Landing page drop-off: investigate if clicks do not become page views.
Qualified lead cost: judge against downstream value, not vanity volume.
CAC: judge only after purchase tracking is trustworthy.
```

## Tracking Rules

Minimum tracking contract:

```text
1. Page view fires on the landing page.
2. Qualified lead event fires only after the right form condition.
3. Wrong-fit lead is not counted as the same conversion.
4. Purchase event is sent server-side from payment or durable status change.
5. Purchase value is dynamic, not hardcoded.
6. Event ID dedupes browser and server events.
7. Email/phone matching data is hashed before sending.
8. Dashboard separates leads, qualified leads, purchases, CAC, and ROAS.
9. Attribution window is chosen intentionally.
10. Recurring revenue is not confused with new acquisition CAC.
```

Compliance rule:

```text
If first-party customer data is shared with an ad platform, privacy policy and consent language must be reviewed before launch.
```

## Reporting Loop

Weekly review:

```text
Export ad data.
Check spend by creative angle.
Check qualified lead cost.
Check purchase signal and CAC.
Check landing page drop-off.
Check whether the winning ad promise matches the landing page.
Check whether sales follow-up handled the leads.
Pick winners, losers, and next creative tests.
```

Do not let the dashboard become the product. The useful report is:

```text
what won
what lost
what broke
what to test next
what workflow leak wastes ad spend
```


```text

Fix the workflow and create/reuse the content production path around it.

```

Discount logic:

```text

Do not discount complexity. Discount retained scope.
```
